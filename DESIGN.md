# DESIGN.md

## 1. 데이터 분석

### 계층 구조 파악

`metadata.json`을 처음 열었을 때 가장 먼저 한 것은 계층 구조를 매핑하는 것이였습니다.

```
단지(전체 배치도)
└── 건물 (101동 / 주민공동시설 / 주차장)
    └── 공종 (건축 / 구조 / 설비 / 소방 ...)
        └── 리비전 (REV1 / REV2 ...)
```

### imageTransform 해석

데이터에서 가장 오래 붙잡혔던 부분이 `imageTransform`이였는데

- `x, y`: 기준 이미지 위의 앵커 포인트 좌표
- `scale`: 오버레이 이미지의 확대/축소 비율
- `rotation`: 회전 각도 (라디안)
- `relativeTo`: 기준이 되는 이미지 파일명

처음에는 `translate(x, y)`로 단순하게 적용했는데 레이어가 전부 엉켰다. 나중에 `x, y`는 단순
offset이 아니라 기준 이미지 위의 앵커 좌표라는 걸 이해한 뒤에 정렬한 것이 맞아 떨어졌습니다.

### 특수 케이스 3가지

데이터를 읽으며 일반 패턴에서 벗어나는 케이스가 3가지였는데,

**케이스 1 — Region 분할 (101동 구조)**

`구조` 공종에 `regions: { A: {...}, B: {...} }` 구조가 있다. 한 공종 내부를 물리적 구역(A/B)으로 나누어 각 구역이 독립된 리비전을 갖는 방식이었기 때문에 이 케이스는 UI에서 "구조 A", "구조 B"를 별도 항목으로 노출하였습니다.

**케이스 2 — 리비전별 독립 imageTransform (주민공동시설 건축)**

일반적으로 `imageTransform`은 공종(discipline) 레벨에 위치한다. 그런데 주민공동시설 건축은 공종 레벨에 `imageTransform`이 없고, 각 리비전(`REV1`, `REV2`, `REV3`)이 독립된 `imageTransform`과 `polygon`을 가진다. 각 리비전이 다른 기준 좌표계를 갖는다는 의미다. 오버레이 렌더링 시 리비전별 transform을 별도로 읽어야 한다.

**케이스 3 — relativeTo 필드의 활용**

`relativeTo`가 다른 두 도면은 공간 좌표계가 달라서 의미 있는 비교가 불가능하다. 이걸 파악하고나서 Compare 모드에서 서로 다른 `relativeTo`를 가진 레이어를 동시 선택하지 못하도록 차단하는 규칙을 추가했습니다.

## 2. 접근 방식

1. **TypeScript 타입 정의 먼저** — `metadata.json` 구조를 `src/types/metadata.ts`에 인터페이스로 모델링했다. 타입을 먼저 확정하면 이후 컴포넌트 작성 시 데이터 접근 패턴이 명확해진다.

2. **Zustand 스토어 설계** — 선택 상태(건물·공종·리비전·뷰 모드·Compare 슬롯)를 스토어에서 정의하고 상태 구조를 먼저 설계하면 컴포넌트 간 데이터 흐름이 자연스럽게 결정

3. **기본 UI 뼈대** — 3패널 레이아웃(사이드바 + 뷰어 + 하단 바)과 도면 트리를 먼저 구현

4. **Single 뷰** — 공종 선택 → 리비전 타임라인 → 이미지 오버레이 파이프라인 구현. 이 단계에서 `imageTransform`을 CSS transform으로 변환하는 방식 선택

5. **Compare 뷰** — 멀티 슬롯 레이어 시스템 구현. 처음에는 단순히 여러 이미지를 겹쳤으나, `relativeTo`가 다른 레이어가 엇나가는 문제를 발견

6. **좌표계 정렬 문제 해결** — `computeRelativeTransformCSS`를 구현하여 slot0 기준의 상대 변환을 계산했다. 각 레이어의 절대 transform을 기준 레이어 기준의 상대 transform으로 변환하는 방식

7. **선택 규칙 강화** — 좌표계 불일치 차단, 빌딩 간 차단, 리비전 비교 모드 규칙을 순차적으로 추가

## 3. UI 설계 결정

### 3패널 레이아웃 (사이드바 + 뷰어 + 하단 컨텍스트)

건물 위치 → 공종 → 리비전의 선택 흐름을 자연스럽게 지원하는 레이아웃을 생각해서 사이드바를 탐색 패널로, 중앙을 뷰어로, 하단을 리비전 타임라인 + 상태 표시로 분리

### 도면 트리 (파일 탐색기 방식)

검색/필터 방식보다 트리 방식을 선택했습니다. 그 이유는 건물 하나에 대한 도면이였기 때문입니다. 추가적으로 도면의 종류가 다양해지고 구역이 많아지면 검색 기능이 필요하다고 판단하였습니다.

### 배치도 건물 폴리곤 클릭

전체 배치도에서 건물 폴리곤을 클릭하면 해당 건물이 선택되고 사이드바가 자동으로 확장되도록하여 직관적으로 구현했습니다.

### Compare 슬롯 + 컬러 코딩

Compare 모드는 최대 4개 레이어를 색상(파랑·주황·초록·보라)으로 구분한다. 투명도 슬라이더로 각 레이어의 가시성을 개별 제어할 수 있다. 오버레이 모드와 나란히 모드를 선택할 수 있어, 정밀 정렬 비교와 전체 비교를 모두 지원했습니다.

### 리비전 타임라인

리비전은 좌→우 시간 순서(구버전→최신)로 배치했다. 클릭 시 `date`, `description`, `changes` 정보를 바로 아래에 표시한다. 변경 내역을 확인하면서 이전/이후를 Compare 슬롯에 올리는 워크플로우가 자연스럽게 연결하였습니다.

### 리비전 비교 모드

동일 공종의 여러 리비전을 Compare 슬롯에 동시에 올릴 수 있는 모드다. 일반 Compare 모드에서는 같은 공종을 여러 번 추가하는 것이 무의미하지만, 리비전 비교에서는 REV1과 REV3를 투명도를 달리해 겹쳐보는 것이 핵심 use case다. 이 두 모드를 구분하고 각각에 맞는 선택 규칙을 적용했습니다.

## 4. 기술 선택

### Zustand

선택 상태(건물·공종·리비전), 뷰 모드, Compare 슬롯 4개 등 여러 글로벌 상태가 서로 연관되어 있어 상태 관리 라이브러리가 필요하였고, Zustand는 스토어를 함수 하나로 정의하고 바로 가져다 쓸 수 있어 처음 접하는 입장에서도 코드 흐름을 이해 용이합니다.

### react-zoom-pan-pinch

도면 뷰어의 핵심 인터랙션(줌·패닝·핀치 제스처)을 최소 코드로 구현하기 위해 선택하여 Canvas 기반 커스텀 구현 대신 이 라이브러리를 사용하면 DOM 레이어 구조(배치도 + SVG 건물 폴리곤 + 이미지 오버레이)를 그대로 유지할 수 있어 CSS transform 기반 정렬과 자연스럽게 연결했습니다.

### Tailwind CSS v4

별도의 CSS 파일 없이 HTML/JSX 안에서 스타일을 직접 작성할 수 있어 컴포넌트 구조와 스타일을 한 곳에서 관리할 수 있기 때문에 선택했습니다.

### Vite 커스텀 미들웨어

`data/drawings/` 폴더의 이미지 파일명이 한글을 포함하기 때문에 브라우저가 URL을 인코딩하면 서버가 파일을 찾지 못하는 문제가 발생하여 Vite의 `configureServer` 훅으로 커스텀 미들웨어를 추가해 `decodeURIComponent`로 경로를 디코딩한 후 파일을 서빙하는 방식으로 해결했습니다.

## 5. 어려웠던 점 및 개선 방안

### 어려웠던 점

**리비전 전환 시 줌 일관성 문제**

공종을 선택하면 도면이 뷰포트에 맞게 표시됐는데, 같은 공종의 다른 리비전으로 전환하면 이미지가 너무 작거나 크게 나오는 문제가 있었습니다.

원인은 처음에 구현했던 방식인 TransformWrapper의 `initialScale={0.15}`는 고정값이라 이미지 크기를 모르는 상태에서 적용되기 때문에 실제로도 구조 A의 두 리비전의 차이가 4배의 차이가 발생하였습니다.

해결 방법으로는 이미지가 로드될 때마다 컨테이너 크기와 이미지 실제 크기를 비교해서 뷰포트에 맞는 스케일을 계산하고 `centerView(fitScale)`로 덮어쓰는 방식을 선택했습니다.

**오버레이 모드 이미지 정렬**

Compare 오버레이 모드에서 두 이미지를 단순히 `position: absolute, top: 0, left: 0`으로 겹쳤더니 이미지가 전혀 맞지 않았다. 각 리비전마다 이미지 해상도가 달랐고, 같은 건물을 서로 다른 크기와 위치로 촬영한 이미지들이었기 때문이다.

예를 들어 주민공동시설 건축의 경우 REV1은 `scale=4.02, x=2481`, REV3은 `scale=4.03, x=2480`으로 미세하게 달랐다.

하지만 두 이미지가 동일한 기준 좌표계에서 정의되어 있다는 점을 활용했다. Layer A의 scale과 rotation으로 기준 좌표계를 역산한 뒤, Layer B를 그 위에 상대적으로 얹는 방식이다. 이 로직을 `computeRelativeTransformCSS(tA, tB)`로 분리해, overlay 렌더링 시 slot0을 tA, 나머지 슬롯을 tB로 넣어 호출했다.

### 개선 방안

- **도면·공종 텍스트 검색** : 도면 수가 늘어날 경우 트리 탐색만으로는 사용자가 원하는 도면을 찾는 것에 대해 어려움을 느낄 것이라고 판단하였기때문에 검색 기능을 추가하여 공종명·리비전 날짜로 등의 필터를 사용하여 빠르게 이동할 수 있도록 개선 필요하다고 판단
- **모바일 레이아웃 최적화** : 현재 3패널 레이아웃은 데스크탑, 태블릿 중심이기 때문에 추가적인 모바일·현장 기기에서의 터치 UX 최적화가 필요하다고 판단
